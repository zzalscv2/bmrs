# Code smell table for `eva_data_analysis.py`

| # | Code smell | Where it occurs | Concrete example | Why this is a smell | Concrete solution |
|---:|---|---|---|---|---|
| 1 | God Script / God Module | Whole file, especially `main()` | `main()` reads data, exports CSV, calculates summaries, sorts data, plots, and controls output files. | One file and one workflow function know too much about unrelated responsibilities. A change to plotting, data cleaning, or file handling can affect the whole program. | Split into modules such as `io.py`, `cleaning.py`, `analysis.py`, `plotting.py`, `config.py`, and `cli.py`. Keep `main()` as a thin coordinator only. |
| 2 | Long / overloaded workflow function | `main(input_file, output_file, duration_by_astronaut_output_file, graph_file)` | `main()` calls reading, exporting, summarising, sorting, plotting, and printing start/end messages. | The function is not very long by line count, but it is long in responsibility. It mixes orchestration with knowledge of all workflow stages. | Replace with smaller orchestration: `load_data()`, `clean_data()`, `export_outputs()`, `generate_reports()`, `generate_plots()`. |
| 3 | Mixed concerns | `read_json_to_dataframe()` | Reads JSON, parses dates, converts `eva` to float, drops missing rows, and prints status. | Reading data and cleaning data are separate reasons to change. Combining them makes testing and reuse harder. | Use `read_eva_json()` only for reading, and `clean_eva_data()` for type conversion and missing-value handling. |
| 4 | Mixed concerns | `plot_cumulative_time_in_space()` | Adds duration hours, calculates cumulative time, plots, saves the image, shows the image, and prints status. | Calculation, visualisation, file output, and user interaction are coupled. This makes the function hard to test without creating files or opening windows. | Split into `calculate_cumulative_eva_time()`, `create_cumulative_plot()`, and `save_or_show_plot()`. |
| 5 | Misleading docstring | `summary_duration_by_astronaut()` | Docstring says it “saves resulting table to a CSV file”, but the function only returns a dataframe. | The documentation describes a side effect that does not happen. This misleads maintainers about the function contract. | Change the docstring to: “Return total EVA duration in hours for each astronaut. This function does not write files.” |
| 6 | Redundant comments | `main()` and `write_dataframe_to_csv()` | `# Read the data from JSON file` appears immediately before `read_json_to_dataframe(input_file)`. | The comment repeats the code rather than explaining intent, constraints, or non-obvious behaviour. | Remove obvious comments. Keep comments only where they explain why something is done. |
| 7 | Misleading / typo-prone comment | `summary_duration_by_astronaut()` | `# dropping the extra 'duration' column as it contains string values not suitable for calulations` | The comment contains a typo and explains an implementation detail that would be clearer if the code were split into named steps. | Replace with a named helper, e.g. `replace_text_duration_with_hours(subset)`, or simply remove the comment after refactoring. |
| 8 | Magic column names | Across the file | `'date'`, `'duration'`, `'eva'`, `'crew'`, `'duration_hours'`, `'cumulative_time'` | Schema knowledge is scattered through the code. A column rename requires many manual edits. | Define constants: `DATE_COLUMN = "date"`, `DURATION_COLUMN = "duration"`, `CREW_COLUMN = "crew"`, etc. |
| 9 | Magic parsing delimiters | `text_to_duration()` and `summary_duration_by_astronaut()` | `duration.split(":")`; `str.split(';')` | The expected data format is hidden in raw string delimiters. | Define constants such as `DURATION_SEPARATOR = ":"` and `CREW_SEPARATOR = ";"`, or wrap parsing in named helper functions. |
| 10 | Magic plotting style | `plot_cumulative_time_in_space()` | `plt.plot(df['date'], df['cumulative_time'], 'ko-')` | `'ko-'` is compact matplotlib syntax, but it hides meaning from readers. | Replace with explicit keyword arguments, e.g. `marker="o"`, `linestyle="-"`, `color="black"`, or move style into configuration. |
| 11 | Magic file paths | `if __name__ == "__main__"` block | `'data/eva-data.json'`, `'results/eva-data.csv'`, `'results/cumulative_eva_graph.png'` | Output locations are embedded in executable code. Changing paths requires code edits. | Move defaults into constants or a configuration object such as `EvaAnalysisConfig`. |
| 12 | Hard-coded encoding | `read_json_to_dataframe()` and `write_dataframe_to_csv()` | JSON uses `encoding='ascii'`; CSV uses `encoding='utf-8'`. | ASCII is brittle for names or metadata containing non-ASCII characters. Encoding choices are hidden in implementation. | Use UTF-8 consistently unless there is a documented reason not to: `encoding="utf-8"`. |
| 13 | Dead code | `calculate_crew_size()` | Function is defined but not used in the workflow. | Dead code increases the amount of code that must be read, tested, and maintained. | Delete it unless crew size is a required output. |
| 14 | Dead code | `add_crew_size_column()` | Function is defined but never called. | It suggests an abandoned feature and depends on `calculate_crew_size()`, which is also unused. | Delete it, or add a deliberate output such as `eva-data-with-crew-size.csv`. |
| 15 | Hidden side effect | `main()` | `eva_data.sort_values('date', inplace=True)` | The dataframe is mutated in place. Later code receives changed data without that change being explicit in the function result. | Use `sorted_eva_data = eva_data.sort_values(DATE_COLUMN).copy()`. |
| 16 | Hidden side effect | `plot_cumulative_time_in_space()` | `plt.savefig(graph_file)` and `plt.show()` | Calling a plotting function unexpectedly writes a file and opens a display window. | Add explicit parameters such as `save_path` and `show=False`, or separate plotting from saving/showing. |
| 17 | Hidden side effect / noisy utility functions | `read_json_to_dataframe()`, `write_dataframe_to_csv()`, `add_crew_size_column()`, `plot_cumulative_time_in_space()` | Each function prints status messages directly. | Low-level functions are harder to reuse in tests, notebooks, services, or GUIs because they always write to the console. | Replace `print()` with `logging`, or keep all user-facing messages in `main()`. |
| 18 | Repeated knowledge | Duration handling across several functions | `dropna(... ['duration', 'date'])`, `df_copy["duration_hours"] = df_copy["duration"].apply(...)`, `subset = df.loc[:, ['crew', 'duration']]` | Several parts of the program independently know that duration is a text column that must be converted. | Centralise duration conversion in `add_duration_hours()` and use constants for all duration-related column names. |
| 19 | Repeated knowledge | Output writing | `write_dataframe_to_csv(eva_data, output_file)` and `write_dataframe_to_csv(duration_by_astronaut_df, duration_by_astronaut_output_file)` | The workflow repeatedly handles CSV output paths manually. | Use a configuration object and an `export_results()` function that knows which output artefacts are produced. |
| 20 | Fragile duration parsing | `text_to_duration()` | `hours, minutes = duration.split(":")` | Invalid data such as `"1:75"`, `"01:05:10"`, blank strings, or non-string values will fail or produce wrong assumptions. | Validate input, strip whitespace, check minute range, and raise a clear `ValueError` with the invalid value. |
| 21 | Fragile crew parsing | `summary_duration_by_astronaut()` | `subset.crew = subset.crew.str.split(';').apply(lambda x: [i for i in x if i.strip()])` | Empty entries are removed, but names are not stripped. This can create separate groups for `"Neil Armstrong"` and `" Neil Armstrong"`. | Use a helper: `split_crew_members(crew_text)` that strips each name and removes blanks. |
| 22 | Poor naming / inaccurate abstraction | `read_json_to_dataframe()` | The function name says “read”, but it also cleans, converts, and drops rows. | The name hides important behaviour. Callers cannot tell that data is being modified. | Rename or split: `read_eva_json()` and `clean_eva_data()`. |
| 23 | Poor naming / inaccurate abstraction | `plot_cumulative_time_in_space()` | The function calculates cumulative time as well as plotting it. | The name suggests display only, but the function also performs analysis. | Split into `calculate_cumulative_time()` and `plot_cumulative_time()`. |
| 24 | Unclear numeric adjustment | `calculate_crew_size()` | `return len(re.split(r';', crew))-1` | The `-1` encodes an assumption about a trailing semicolon, but this is not documented in the code. | Parse crew members properly, then return `len(split_crew_members(crew))`. |
| 25 | Unnecessary dependency | Top-level import | `import re` is only used by dead crew-size code. | If the dead code is removed, the import becomes unnecessary. | Delete `import re` after removing `calculate_crew_size()`. |
| 26 | Weak command-line handling | `if __name__ == "__main__"` block | `if len(sys.argv) < 3:` then manually reads `sys.argv[1]` and `sys.argv[2]`. | Manual argument parsing is brittle and gives poor help/error messages. | Use `argparse` with named arguments: `--input`, `--cleaned-csv`, `--summary-csv`, `--graph`, `--show-plot`. |
| 27 | Inconsistent output configuration | `if __name__ == "__main__"` block | Input and main CSV can be customised, but graph and astronaut summary paths are always hard-coded. | Users can only partly configure the workflow. This is surprising and inflexible. | Expose all output paths through command-line arguments or a config object. |
| 28 | Inconsistent data ownership | `summary_duration_by_astronaut()` | `subset = df.loc[:, ['crew', 'duration']]` followed by `subset.crew = ...` | This may trigger pandas view/copy ambiguity and makes it unclear whether the original dataframe could be affected. | Use `subset = df[[CREW_COLUMN, DURATION_COLUMN]].copy()` before modifying it. |
| 29 | Poor testability | `plot_cumulative_time_in_space()` | The function depends on global `matplotlib.pyplot` state and calls `plt.show()`. | Automated tests should not need to open a plot window or rely on global plotting state. | Return a `Figure`/`Axes` object from a plotting function, then save/show at the application boundary. |
| 30 | Poor testability | `main()` | `main()` performs the full end-to-end workflow with real files and plotting. | Unit tests would need real input/output files and may produce external artefacts. | Test pure functions separately and keep `main()` minimal. Use temporary directories for integration tests only. |
| 31 | Lack of validation before output | `write_dataframe_to_csv()` | Writes to paths such as `results/eva-data.csv` without ensuring the directory exists. | The program may fail if the `results` directory has not been created. | Before writing, call `Path(output_file).parent.mkdir(parents=True, exist_ok=True)`. |
| 32 | Low-level function has application-specific wording | `write_dataframe_to_csv()` | `print(f'Saving to CSV file {output_file}')` | A general CSV-writing helper should not decide user-facing wording. | Remove printing from the helper, or log at the workflow level. |
| 33 | Dataframe schema not explicit | Whole file | The code assumes columns such as `crew`, `duration`, `date`, and `eva` exist. | Missing or renamed columns will cause runtime errors at scattered locations. | Add `validate_eva_schema(df)` to check required columns once after loading. |
| 34 | Weak error reporting | `text_to_duration()` and `read_json_to_dataframe()` | Parsing or missing columns will raise raw pandas/Python errors. | The user may receive a low-level error without context about the EVA data problem. | Raise domain-specific messages such as `ValueError("Expected duration in HH:MM format")`. |
| 35 | Unclosed plot state | `plot_cumulative_time_in_space()` | `plt.savefig(graph_file)` and `plt.show()` are called, but `plt.close()` is not. | Repeated runs can accumulate matplotlib state and memory usage. | Use `fig, ax = plt.subplots()` and call `plt.close(fig)` after saving/showing. |
| 36 | Inconsistent return style | Several functions | Some functions return dataframes; others write files or display plots and return `None`. | Mixed return and side-effect styles make the code harder to reason about. | Prefer pure transformation functions that return data; keep side effects in I/O functions. |
| 37 | Tight coupling to pandas | Most functions | Business rules such as duration conversion and crew splitting are embedded inside dataframe operations. | Simple logic is harder to test without constructing dataframes. | Keep small pure helpers such as `text_to_duration_hours()` and `split_crew_members()` independent of pandas. |
| 38 | Lack of configuration object | `main()` parameters and `__main__` block | Four loose path parameters are passed into `main()`. | Related settings are not grouped, so function signatures grow as options increase. | Introduce `@dataclass EvaAnalysisConfig` containing input path, output paths, encoding, and plot options. |
| 39 | Unclear ordering dependency | `main()` and `plot_cumulative_time_in_space()` | `main()` sorts by date before calling the plot function, while the plot function assumes input is already sorted. | Correct plotting depends on a hidden precondition outside the function. | Move sorting into `calculate_cumulative_eva_time()` so the calculation always sorts before `cumsum()`. |
| 40 | Implicit index behaviour | `summary_duration_by_astronaut()` | `subset = subset.groupby('crew').sum()` | The astronaut name becomes the dataframe index, which may be unexpected when exporting or reusing the result. | Use `groupby(CREW_COLUMN, as_index=False)` so the astronaut remains an ordinary column. |

## Prioritised fixes

The highest-value changes are:

1. Split reading, cleaning, analysis, plotting, and command-line handling.
2. Replace repeated string literals with constants.
3. Remove dead crew-size code unless it is required.
4. Make duration and crew parsing explicit, validated, and testable.
5. Remove hidden side effects such as `inplace=True`, `plt.show()`, and printing inside utility functions.
6. Replace manual `sys.argv` handling with `argparse`.
7. Add schema validation and clearer error messages.

These changes would make the script easier to understand, safer to modify, easier to test, and more reusable in other contexts.
