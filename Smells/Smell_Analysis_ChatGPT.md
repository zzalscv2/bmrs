# Code Smell Analysis by ChatGPT: `eva_data_analysis.py`

## Basis of analysis

This framework applies the uploaded `Smells.md` taxonomy to `eva_data_analysis.py`. A code smell is treated as an indicator of a deeper design, implementation, or maintainability problem rather than necessarily as a functional defect. The analysis is ordered by source line number.

## Summary

- Total smell occurrences identified: **79**.

- Most frequent themes: hard-coded configuration and schema values, hidden side effects, repeated parsing/conversion knowledge, weak separation between data transformation and I/O, and misleading/incomplete comments.

- The most important refactoring direction is to separate the pipeline into loading, validation/cleaning, enrichment, analysis, plotting, and persistence boundaries.

## Consolidated refactoring framework

1. **Introduce schema constants** for `date`, `duration`, `duration_hours`, `crew`, `crew_size`, and `cumulative_time`. This removes repeated magic strings and improves analysability.

2. **Create an `EvaAnalysisConfig` dataclass** containing input, cleaned CSV, astronaut summary CSV, graph output, encoding, plot labels, plot style, and whether to show the plot. This addresses data clumps, long parameter lists, hard-coded settings, and hard-coded paths.

3. **Separate pure transformations from side effects**. Loading, saving, printing/logging, saving plots, and showing plots should be outer-boundary operations; parsing, cleaning, and aggregation should be pure functions that return new dataframes.

4. **Centralise domain parsing**. Implement one `parse_duration_hours` and one `parse_crew_members` function. Reuse them in crew-size calculation, astronaut summary, and cumulative plotting.

5. **Replace manual CLI handling with `argparse`**. This gives explicit defaults, help text, validation, and avoids direct dependence on `sys.argv`.

6. **Make plotting non-interactive by default**. Return a figure from the plotting function and save/show it only when requested by configuration. This improves testability and headless execution.

7. **Remove or integrate unused functionality**. `add_crew_size_column` should either become part of the enrichment pipeline or be removed.

8. **Revise comments and docstrings** so they describe actual behaviour, especially where functions clean data, return summaries, or perform I/O.
