# Maintainability Analysis of `eva_data_analysis.py` against ISO/IEC 25010:2023

## Basis of analysis

This analysis uses the maintainability characteristic and sub-characteristics defined in `ISO250102023.md`:

- **Maintainability**: capability of a product to be modified by the intended maintainers with effectiveness and efficiency.
- **Modularity**: capability of a product to limit changes to one component from affecting other components.
- **Reusability**: capability of a product to be used as assets in more than one system, or in building other assets.
- **Analysability**: capability of a product to be effectively and efficiently assessed regarding the impact of intended changes, diagnosed for deficiencies or causes of failures, or used to identify parts to be modified.
- **Modifiability**: capability of a product to be effectively and efficiently modified without introducing defects or degrading existing product quality.
- **Testability**: capability of a product to enable an objective and feasible test to be designed and performed to determine whether a requirement is met.

## Summary judgement

`eva_data_analysis.py` is small and partly decomposed into functions, which gives it some initial maintainability. However, the implementation still presents several maintainability violations and threats. The main issues are mixed responsibilities, hidden side effects, hard-coded paths and output locations, weak validation, fragile parsing assumptions, direct plotting and display inside processing code, unstructured command-line handling, and dataframe mutation without clear contracts. These issues reduce modularity, analysability, modifiability, reusability, and testability.

The most serious maintainability threats are:

1. `main()` coordinates reading, exporting, summarising, sorting, plotting, and status output without separating orchestration from data processing.
2. Several functions combine computation with I/O, display, logging, or mutation.
3. Important assumptions about file paths, data schema, duration format, crew formatting, encodings, and output directories are implicit rather than validated.
4. Some functions mutate dataframes or create derived columns without a clear and consistent policy.
5. There is no objective test boundary for the end-to-end behaviour because dependencies such as files, `sys.argv`, `matplotlib`, and printed output are directly embedded.

## Recommended refactoring direction

A maintainable redesign should separate the script into clearer responsibilities:

```text
eva/
  cli.py                 # argparse, user-facing command-line interface
  io.py                  # read/write CSV/JSON and path handling
  validation.py          # schema and value validation
  duration.py            # duration parsing and conversion
  crew.py                # crew parsing and crew-size logic
  transforms.py          # dataframe transformations
  summaries.py           # astronaut-level aggregation
  plotting.py            # figure construction and optional saving
  pipeline.py            # orchestration of the analysis workflow
  tests/                 # unit and integration tests
```

The immediate improvements should be:

1. Replace manual `sys.argv` parsing with `argparse`.
2. Remove `print()` calls from core processing functions and use logging only at the CLI layer.
3. Remove `inplace=True` and adopt a consistent rule that transformations return new dataframes.
4. Split reading from cleaning and validation.
5. Extract crew parsing and duration parsing into pure, independently tested functions.
6. Separate cumulative-time calculation from plot construction, file saving, and interactive display.
7. Parameterise all file paths and create output directories safely.
8. Add tests for duration parsing, crew parsing, cleaning, aggregation, cumulative-time calculation, and CLI argument handling.

## Overall maintainability conclusion

The script is understandable because it is short, but it is not strongly maintainable under ISO/IEC 25010:2023. Its main maintainability risk is not size alone, but the way responsibilities, assumptions, side effects, and data transformations are mixed. The highest-priority changes are to separate pure data transformations from I/O and plotting, make schema assumptions explicit, remove hidden mutation, and introduce focused tests around each transformation step.
